## ADR Tag Vocabulary

This document defines the **controlled vocabulary of tags** used in the Architectural Decision Records (ADR) of this repository.

Tags allow:

- consistent categorization of ADRs
- easier navigation across architectural decisions
- improved indexing by search tools
- better interpretation by AI coding agents

All ADRs should use tags defined in this document.

---

## Tag Categories

Tags are grouped into several conceptual categories:

- Architecture
- Integration
- Processing
- Infrastructure
- Operational
- Development

These categories exist for documentation purposes and do **not** need to appear as tags themselves.

---

## Architecture Tags

### hexagonal-architecture

Indicates that the ADR defines or affects the system’s hexagonal architecture.

Used when discussing:

- ports and adapters
- domain boundaries
- architectural layering

---

### ports-and-adapters

Relates specifically to the **Ports & Adapters architectural pattern**.

Used when defining:

- inbound ports
- outbound ports
- adapter responsibilities

---

### system-architecture

Used for decisions affecting the architecture of the entire system.

Examples:

- architectural patterns
- dependency rules
- system layering

---

### domain-boundaries

Defines separation between:

- domain layer
- application layer
- infrastructure layer

---

### project-structure

Used when defining the layout of the repository or codebase.

Examples:

- directory structure
- module organization
- project layout

---

## Integration Tags

### external-integration

Indicates interaction with an external service or system.

Examples:

- third-party APIs
- message providers
- external platforms

---

### gmail-api

Used for ADRs related specifically to Gmail integration.

Examples:

- message retrieval
- Gmail labeling
- Gmail adapters

---

### infrastructure-adapter

Used when defining adapters connecting the system to infrastructure.

Examples:

- API clients
- filesystem adapters
- messaging adapters

---

## Processing Tags

### message-processing

Used for ADRs that define the message processing pipeline.

Examples:

- processing workflows
- pipeline stages
- event extraction

---

### parsing

Used for ADRs related to message parsing.

Examples:

- parsing strategies
- parser implementations
- message interpretation

---

### parser-registry

Used when defining mechanisms to register or resolve parsers.

Examples:

- parser lookup
- parser registration

---

### parser-selection

Used when defining how the correct parser is selected.

Examples:

- classification-based dispatch
- parser routing strategies

---

### message-classification

Used when defining message classification logic.

Examples:

- identifying the issuing bank
- identifying notification types

---

## Infrastructure Tags

### outbound-port

Used when defining outbound ports in the hexagonal architecture.

Examples:

- persistence ports
- state management ports

---

### storage

Used for decisions about data persistence or artifact storage.

Examples:

- filesystem storage
- database persistence

---

### state-management

Used when defining how the system tracks operational state.

Examples:

- message processing state
- workflow state

---

## Operational Tags

### observability

Used for ADRs related to system visibility.

Examples:

- logging
- debugging artifacts
- monitoring

---

### data-retention

Used when defining policies about data lifetime.

Examples:

- artifact retention
- log rotation

---

## Development Tags

### dependency-injection

Used when defining how dependencies are provided to components.

Examples:

- DI strategies
- service injection

---

### composition-root

Used when defining where dependencies are wired together.

Examples:

- bootstrap modules
- application startup configuration

---

### wiring

Used for decisions about how components connect inside the system.

Examples:

- service wiring
- adapter binding

---

## Tag Usage Rules

### Rule 1 — Use only defined tags

ADRs should use tags defined in this document whenever possible.

---

### Rule 2 — Limit tag count

Each ADR should typically include **3–5 tags**.

---

### Rule 3 — Prefer specific tags

Prefer precise tags over generic ones.

Example:

Use:

```

parser-registry

```

instead of:

```

parsing

```

when the decision specifically concerns parser registry behavior.

---

### Rule 4 — Combine tag types

A typical ADR should include tags from multiple categories.

Example:

```yaml
tags:
  - parser-registry
  - message-processing
  - domain-logic
```
